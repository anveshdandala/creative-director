Authentication Flow

AI Content Director — Clerk sign-in + own JWT handoff

Phase 1 — Sign-in (happens once per session)
1
Browser User signs in via Clerk UI
Clerk handles the actual credential check (email/password, Google, GitHub, etc). Your app never sees a password.
↓
2
Clerk Issues a short-lived session JWT
Clerk gives the browser a session token (~60s expiry). It contains only the Clerk user ID (sub) — not email, not name.
↓
3
Browser useAppAuth hook calls syncUser()
The useAppAuth hook detects sign-in, gets the Clerk token via getToken(), and immediately POSTs it to the backend.
POST /api/auth/sync
Authorization:Bearer <clerk_session_token>
↓
4
Backend Verifies Clerk token + fetches real user data
verifyToken() checks the Clerk JWT signature using CLERK_SECRET_KEY. Then calls clerk.users.getUser(clerkId) to get the real email and name (not in the token by default).
↓
5
Prisma Postgres Upserts user row
Creates the User record if first sign-in, or updates email/name if returning. The clerkId is stored once here and never used again downstream.
↓
6
Backend Issues our own JWT
Signs a new JWT with JWT_SECRET. Payload is just { sub: user.id } — our internal Postgres ID, not the Clerk ID.
Response
token:<our_jwt> (expires in 7d)
user:{ id, email, name, createdAt }
↓
7
Browser Stores our JWT in localStorage
storeToken(token) writes it under the key app_token. The Clerk token is discarded — Clerk is never used again for API calls.
Phase 2 — Every protected API call (guide, validate, etc.)
8
Browser Attaches our JWT to every fetch
useAppAuth exposes apiHeaders — a pre-built headers object you spread into every fetch call.
e.g. POST /api/guide
Authorization:Bearer <our_jwt>
Content-Type:application/json
↓
9
Next.js Route Handler Forwards token to backend
getOwnToken(req) reads the Authorization header and passes it straight through to the Express backend. No Clerk involved.
↓
10
Backend authenticate middleware verifies our JWT
Calls jwt.verify(token, JWT_SECRET). Looks up payload.sub in Postgres. Attaches req.user = { id, email, name } and calls next().
↓
11
Backend Controller runs with req.user populated
generateGuide / validateDraft read req.user.id to associate the AI generation with the correct user row in Postgres.
Phase 3 — Return visit (already has a token)
12
Browser Skips sync entirely
useAppAuth checks localStorage on load. If app_token already exists, sync is skipped and the stored JWT is used immediately. Sync only runs again after sign-out clears it.
Key design rules
Clerk is a login UI, not an auth layer. It handles credential verification and issues a short-lived token. The moment our backend verifies that token once, Clerk is done. Every subsequent request uses our own JWT — Clerk's SDK is never called again per request.
The JWT payload is minimal. It contains only { sub: user.id } — our internal Postgres CUID. The Clerk ID, email, and name are never in the token. They are loaded fresh from the DB on every request by the authenticate middleware.
No session storage server-side. The backend is stateless — it verifies the JWT signature + DB lookup on every call. No Redis, no session table.
